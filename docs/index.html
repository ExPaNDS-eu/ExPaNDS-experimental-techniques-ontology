Scope of PaNET

PaNET is a collection of experimental techniques that match the following criteria:

  • The technique MUST be in use, or has been used in the past, at a large European photon or neutron facility. Facility-members of LEAPS and LENS are examples of such facilities, as are ESFRI-recognised facilities such as ELI.
  • The technique MUST take advantage of one of the facility's beamlines. A technique is not eligible merely for taking place within a facility's campus.
  • In general, the technique SHOULD NOT distinguish itself from other techniques only by the choice of sample or the sample's state. This criterion may be overruled if there is a very strong view from the user community that the sample is fundamental to the technique definition.
  • A technique MAY distinguish itself from another technique by an essential dependency on some well-defined sample interaction. A laser pump-probe is an example of such an interaction.

PaNET build environment

This repo contains tools for working with the Photon and Neutron Experimental Techniques (PaNET) github repository. In particular when building the ontology file PaNET.owl from the various the input files.

There are two main parts:

  1. A script, build-panet, that builds the PaNET. To work, it requires that various software packages are installed.
  2. A 'Dockerfile' script that describes how to build a container image that contains the build-panet script and the software packages it needs.

Using the script directly

The script 'build-panet' uses:

  • the OBO Robot tool to build the PaNET ontology. Currently, this must be (precisely) v1.8.1: newer versions currently do not work. Note that robot requires that Java is installed.
  • the git command. This is used to deduce a version number for the ontology. It is possible to avoid a dependency on git by specifying the version explicitly; however, this is not recommended.

The script takes an optional argument: the path of a comma-separated values (csv) input file. The script will create a corresponding OWL file (in RDF/XML) in the same directory. The steps involved are:

  1. Generate an initial set of definitions from the CSV file.
  2. Run a reasoner to deduce more information about terms.
  3. Merge some (fairly constant) information about the ontology itself from the 'PaNET_metadata.ttl' file. This is located in the same directory as the CSV file.

If no argument is given then the path source/PaNET.csv is used. This allows the script to be run from the base directory of the PaNET git repository.

Container image

A container image allows you to build the PaNET ontology without installing any dependencies. You only need the ability to run run a container. This may also be useful when automating, as CI/CD frameworks often support containers.

In order to use the containerised PaNET-build script you must first get the container image onto you computer. There are two main ways of doing this: downloading it and building it yourself. These different approaches are described in the next two sections. If you are not sure, follow the instructions for downloading the image.

Once you have the container image, you can use it to build the PaNET ontology. This is also described below.

Downloading the container image

A pre-build container image is available from this git repo's container registry.

Container images have the label panet-build, with the latest version being panet-build:latest. Therefore, the full reference for the latest container version is gitlab.desy.de:5555/paul.millar/panet-build:latest.

Here is an example command showing how to download the latest image with docker:

docker pull gitlab.desy.de:5555/paul.millar/panet-build:latest

Here is the equivalent command when using podman:

podman pull gitlab.desy.de:5555/paul.millar/panet-build:latest

Building the container image

This step is not necessary if you simply want to build the PaNET ontology. Rather, this information is useful if you wish to enhance the build process in a way that requires additional tools.

This git repo contains the Dockerfile file that contains instructions for building the container image.

The image may be built using a standard procedure. The following instructions show how to build the image and tag it panet-build:latest if the repo is the current directory.

Here is the command for docker:

docker build . -t panet-build

and for podman:

podman build . -t panet-build

Using the container image

By default, the container image expects the PaNET git repository to be available as the /work directory within the container image. Typically, this is achieved by "mounting" the PaNET git repository directory (on the host computer) at this location.

This may be done by explicitly specifying the git location.

A typical command with docker is:

docker run -v ~/git/PaNET/:/work --rm -it panet-build

... or with podman:

podman run -v ~/git/PaNET/:/work --rm -it panet-build

Often, the current directory is the PaNET git repository's base directory. In this case, then a somewhat simplified version of the command may be used.

Here is a complete example, showing how to check out PaNET and use docker to build the ontology:

cd ~/git git clone https://github.com/ExPaNDS-eu/ExPaNDS-experimental-techniques-ontology PaNET cd PaNET docker run -v $PWD/:/work --rm -it panet-build

Here is the same example but using podman instead:

cd ~/git git clone https://github.com/ExPaNDS-eu/ExPaNDS-experimental-techniques-ontology PaNET cd PaNET podman run -v $PWD/:/work --rm -it panet-build

The container accepts argument and passes these to the script. Therefore, it is possible to specify a target CSV file, similar to running the script directly. Note that the build process assumes that the ontology metadata is available as the file PaNET_metadata.ttl in the same directory as the source CSV file.

Specifying an alternative source file may be useful, under some specific circumstances. However, the reader is encouraged to edit the PaNET.csv file directly and to use git to manage those changes.

PaNET Standard Operating Procedures

General procedure for handling issues in PaNET

Changes to PaNET ontology are triggered by resolving an issue in github. Other than in exceptional circumstances, it should be possible to trace any change in the PaNET ontology back to a corresponding issue that is being addressed.

The issue is created by someone who has found (or believes they have found) a problem with PaNET. The issue's description (provided by the person submitting the issue) indicates what problem they are trying to resolve. The description may propose a change to the ontology (e.g., add a new term, retire a term). Any such proposal change should be taken as a hint: the important part is to resolve the underlying problem, which may be best achieved by doing something other than what is proposed.

The issue is assigned to someone who is responsible for that issue: the issue resolver. That person is responsible for closing that ticket. Should the person be unable to do that (e.g., currently insufficient spare time, left the facility) then another person should be assigned.

The issue resolver should look at the description and try to understand the underlying problem. Often the description is incomplete and doesn't provide sufficient information to fully understand the underlying problem. If so, the issue resolver should pose questions (through the github issue) to better understand the problem.

Once the problem is understood, there should be a decision whether the underlying problem is something that fits within PaNET's remit. If this is unclear, the issuer resolver may seek a consensus opinion from the PaNET maintenance team (via email or at the next meeting).

If the underlying problem does not fit within PaNET's remit then the issue resolver will update the issue, writing a description of why the problem falls outside PaNET's remit, and close the issue.

If the underlying problem fits within PaNET's remit then the issue resolver should understand how best to resolve the problem. Any suggestion (in the issue description) may be considered, but other possibilities should also be considered. If this is unclear then the issue resolver may discuss this with other members of the PaNET maintenance team (either via mailing list or at a meeting).

Once the desired change is understood, the issue resolver will follow one or more of the procedures (described below) to update the ontology.

The issue will be closed automatically once the ontology is updated.

Naming convention

In PaNET we aim for consistent labels for terms. Therefore, when adding a new term or updating an existing term the labels should follow the naming convention described here.

The following rules were taken from Steve's comment:

  • Names are copied from existing sources wherever possible.
  • Adopted style is lower-case except for acronyms
  • We assume the names are not case-sensitive in order to avoid a proliferation of terms using different cases.
  • We assume that there is no significant preference for label over alt-label, i.e. all labels have the same importance.
  • Abbreviations and acronyms are treated as any other label/alt-label

Adding a new term/relationship

New terms or relationships are added to fix an inadequacy with the current ontology. Examples of such problems include the inability to make a distinction between two similar techniques when searching, or the inability to query a well-defined concept that encompasses multiple terms.

The motivation for adding a new term usually comes from following the "General procedure for handling issues in PaNET" (see above).

The issue resolver will need to understand the exact nature of the new term or the new relationship. For terms, this involves writing an appropriate description, identifying the relationships with existing PaNET terms, and potentially finding links to related terms in other corpora. This process may involve contacting experts in the field to try to ensure a consensus in how this term is being defined.

As a general guideline, a new term should be a subclass of the classes that are the conceptually closest broader techniques. If any of these closest broader techniques classes have existing subclasses then those relationships are reviewed to see whether the new term should be added as an intermediate term between one of the closest broader techniques and the existing subclass.

In case the closest broader technique has a relation to a technique where the new term is when added the closest broader technique, the relation between the existing classes should be removed and the new technique should be put in between. To give a more concrete example, if PaNET contains two techniques A and B with A being a generalisation of B (in this example, written as A ← B) and a new term N is introduced where A ← N and N ← B then the existing relationship (A ← B) is removed when the two new relationships (A ← N and N ← B) are added.

Once the new term or relationship is understood, the issue resolver will update PaNET ontology following the general procedure for updating git, as described below.

Removing a term

A problematic term may be removed ONLY if it has not been released. In other words, a term may be removed only if it was added more recently than the most recent release.

If the term is present in one or more releases then the problematic term must be retired instead. See below, in the "Retiring a term" section, for more details.

Removing a term follows the procedure described in "general procedure for updating git" with the slight modification that the patch may be generated automatically, using the git revert command, rather than manually editing the corresponding files.

Care must be exercised when using the git revert command as it undoes everything done by a specific commit. If that commit does more than adding a term we want to get rid of, then an alternative approach may be needed.

Also, be aware that changes in the ontology after the new term was added may complicate the process of removing a term. This is most obvious if the git revert command indicates a conflict that must be resolved, but this is a more general problem. For example, if a term (term #1) is added that is subsequently discovered to be problematic and a later commit adds a new term (term #2) that is a child of term #1 then the commit removing term #1 will also need to handle the parent-child relationship between term #1 and term #2. Simply reverting the earlier patch that introduced term #1 (i.e., git revert) will not do this.

Retiring a term

This process is based on Geneontology's Principles for term obsoletion.

A term is retired because its presence is causing a problem in the PaNET ontology. For example, the term may be badly defined, allowing multiple mutually incompatible interpretations on to what the term refers.

Released terms are never removed.

More specifically, the PaNET maintenance team guarantees that any IRI corresponding to a technique that is available in one release will exist in all subsequent releases (see below on releasing).

Retiring a term is the process of adhering to this promise while, at the same time, acknowledging that a specific term should not be used any more.

Retiring a term involves removing all its links within the ontology and adding an annotation to indicate the term is obsolete.

In some cases it is desirable to "simultaneously" retire a term while creating a new term. An example is when an existing term is badly defined: the existing term (with a bad definition) is retired while a new term (with a better definition) is added. Such a change in behaviour should be done as a single git commit, in order to indicate the close connection between these two activities.

Once the desired changes are understood, the issue resolver updates the git repo, as described in the "general procedure for updating git" section below.

Removing or modifying a relationship

Unlike terms, relationships are not guaranteed from release to release: they may be modified, as we correct problems and introduce new terms. Therefore, any problematic connections may be simply modified or removed, as appropriate.

Any changes to the relationship should be motivated by solving some concrete problem. That problem should be described in a GitHub issue. Ideally, the issue should describe some user-facing problem (e.g., when searching for X, I receive results like Y even those they are not appropriate). In practice, when the underlying problem is obvious, the issue may state simply that the relationship is wrong.

There are situations where multiple changes are necessary; for example, if a relationship is wrong then fixing the problem may involve removing the existing relationship and adding a new relationship. When multiple changes are motivated from solving the same underlying problem then those changes should be provided as a single git commit.

Once the problem is suitably investigated and the desired modification to the ontology is well understood, general procedure for updating git is followed.

Making a Release

This section describes the procedure for making a new PaNET release.

A release represents a stable, reliable version of the evolution of PaNET: a point-in-time snapshot of the ontology. Releases are given a human-friendly name, involving three numbers; e.g., v1.5.2.

Releases are the outwardly visible evolution of the PaNET ontology. External users of the ontology are encouraged to depend only on released versions of PaNET. The strong guarantees of backwards compatibility apply only to released versions of PaNET and not on any intermediate (internal) non-released ontologies.

A release can NEVER be changed or modified in any way, even if problems are subsequently discovered, even if those problems are catastrophic in nature. If problems are discovered then the release notes should be updated to reflect those problems. If catastrophic problems are discovered then the release notes should indicate that we strongly discourage people from using that version. Problems may also be notified on the announcements mailing list.

Release Procedure

The following procedure describes how to make a release:

1. Pre-release Checks

Check the CI/CD ("Continuous Integration and Deployment", the automatic testing that GitHub does, to verify there are no problems; look for a green tick on the main code page) to ensure it is not indicating any problems. You should also look at the Github issues to see if any release-critical problems have been identified.

2. Repository Setup

Ensure you have an up-to-date clone of the PaNET git repository. If you do not already have a git repo then clone it. Note that the following command should be a single line:

git clone git@github.com:ExPaNDS-eu/ExPaNDS-experimental-techniques-ontology.git

If you have already cloned the github repo then ensure it is up-to-date with the following two commands:

git checkout master git pull --ff-only git status

3. Create Change Log

Write a list of what has changed in this release (compared to the previous release): a "change log".

The authoritative source of information on what has changed is the list of git commits, those since the most recent release (i.e., the current PaNET version). The following command provides a one-line summary of all the changes since the last release:

git log --pretty=oneline CURRENT_VERSION..

For more information, use the "short" output:

git log --pretty=short CURRENT_VERSION..

For example, if you are working on releasing PaNET version 1.2.0 then the most recent release would be 1.1.0 (in other words, the current version is 1.1.0). This release would be identified by the v1.1.0 tag. The following command provides the one-line summary of what has changed:

git log --pretty=oneline v1.1.0..

The latest tag is found with the git tag command

Update the existing CHANGELOG file with information about the latest release at the top. Use the previous release as a template.

4. Build PaNET

panet-build

or, the following command if using the container image with docker:

docker run -v $PWD:/work --rm -it panet-build

or the following command if using the container image with podman:

podman run -v $PWD:/work --rm -it panet-build

See the panet-build repo for more details on how to build PaNET.

When building PaNET, keep an eye on the output. Look for any messages that might indicate a problem.

5. Verification

As far as possible, verify that PaNET is correct. For example:

Manually inspect the generated OWL file:

less source/PaNET.owl

Does it have the correct version information?

Look at the generated documentation by opening the directory widoco in your favourite web-browser. Does it look "reasonable". Is the version information correct?

This is your LAST CHANCE to spot any problems with the release!

6. Create Release Commit

Create a commit that updates the version information. Note that the following command should be a single line:

git commit -m "Preparing for release" CHANGELOG.md

7. Tag the Release

Create a tag for the release:

git tag VERSION-IDENTIFIER

where VERSION-IDENTIFIER is a lower-case v followed by the version number. For example, if you are releasing version 1.5.2 then the command would be:

git tag v1.5.2

Take a deep breath, you are about to make history!

(Up to now, any mistakes or problems can be fixed. The next step makes the release indelible.)

8. Publish the Release

Push the changes to github, including the tag:

git push --tags origin master

This command will fail if anyone has modified the master branch since you updated your local repo (see step 2.). If this happens, then:

  • Remove the tag: git tag -d VERSION-IDENTIFIER
  • Discard your commits: git reset --hard origin/master
  • Go to step 1.

9. Post-Release Verification

Verify that the CI/CD has correctly built the release, and that the new version is available through the persistent URL. This may take a few minutes.

Update github releases: Draft new release, Download artefacts from CI/CD job. Unzip contents to extract the two files: PaNET.owl PaNET_reasoned.owl. Upload these two files into the release. Make the release public.

  • https://bioportal.bioontology.org/ontologies/PANET
  • https://www.bioontology.org/wiki/BioPortal_Help#Updating_an_existing_ontology_submission

It seems it is possible to have automatic update (to be confirmed)

Verify that the purl.org URL is pointing to the latest version; e.g.,

paul@celebrimbor:~$ curl -s -L http://purl.org/pan-science/PaNET/PaNET.owl | grep versionInfo <owl:versionInfo rdf:datatype="http://www.w3.org/2001/XMLSchema#string">1.2.0</owl:versionInfo> paul@celebrimbor:~$

10. Announcement

Write an email to the <panet-announce@desy.de> mailing list that informs people of the new version and what has changed.

Congratulations, you have successfully released PaNET.

Git Workflow

PANET development follows a structured Git workflow to ensure code quality and collaboration efficiency.

General procedure for updating git

There are several kinds of changes one may make to the PaNET git repository, motivated by resolving different kinds of problems. They should all follow the following procedure:

1. Create a pull request:

a. Ensure you have an up-to-date clone of PaNET git repo.

If you do not already have a git repo then you should clone it. The following example should be a single command, but it split for readability.

git clone git@github.com:ExPaNDS-eu/ExPaNDS-experimental-techniques-ontology.git

If you have already cloned the github repo then ensure it is up-to-date:

git checkout master git pull --ff-only

b. Create a new branch within your local git repo and "check out" (start using) that new branch.

The branch's name should be chosen based on the github issue it is addressing:

git checkout -b <branch> origin/master

For example, the branch could be called ISS-51-add-term-tachyon-spectroscopy when resolving Github issue 51:

git checkout -b ISS-51-add-term-tachyon-spectroscopy origin/master

c. Make appropriate changes to the files according to the desired change:

emacs source/PaNET.csv

Use the appropriate command if your favourite editor isn't Emacs. You may find the file easier to edit when using a spreadsheet; e.g., Libreoffice's calc.

d. Build PaNET:

panet-build

or, the following command if using the container image with docker:

docker run -v $PWD:/work --rm -it panet-build

or the following command if using the container image with podman:

podman run -v $PWD:/work --rm -it panet-build

See the panet-build repo for more details on how to build PaNET.

When building PaNET, keep an eye on the output. Look for any messages that might indicate a problem.

e. As far as possible, verify the changes have the desired effect. Here are some suggestions:

Manually inspect the generated OWL file:

less source/PaNET.owl

Look at the generated documentation by opening the directory widoco in your favourite web-browser.

If you find a problem, then try to resolve it by going back to step c.

f. Create a commit that includes those changes.

The commit message should indicate that the commit closes the corresponding issue:

Add each modified files using the git add command:

git add source/PaNET.csv

Create the commit:

git commit

This command will open an editor for you to write the commit message. The first line of the commit message is a single-line summary of what has changed. As a convenience, the first word of the first line should be the issue identifier (e.g., ISS-51 for issue 51).

The following is an example of a minimal commit message:

ISS-51 Add new term tachyon spectroscopy Closes: #51

Here is an example commit message that follows the motivation-modification-result form, which is best practice in some software development projects:

ISS-51 Add new term tachyon spectroscopy Motivation: Ontology currently doesn't allow it to distinguish spectroscopy based on tachyon emission. Modification: Add a new term, a child of emission spectroscopy, that represents this specialised form of spectroscopy. Result: It is now possible to identify tachyon emission spectroscopy. Closes: #51

The phrase "closes: #nn" tells GitHub that committing this change fixes the underlying problem described in the corresponding issue. GitHub will update the issue with information related to the commit and automatically close the issue when the commit is received by GitHub.

g. Push the branch to PaNET git repo:

git push origin HEAD

h. Create a pull request that targets the master branch.

This involves using the web-browser:

  • When pushing a branch to github, the server will respond with a message that provides the URL for creating the pull request.
  • Open this URL in your favourite web browser.
  • Verify the information on the page is correct; in particular, that the remote branch will be deleted once the pull request is accepted.
  • Click the green Create pull request button.

2. Receive agreement on proposed changes.

  • Any pull request requires at least one team member to "sign off" on the change.
  • Try to identify someone who will review your changes.
  • Work with them to get their agreement that the patch should go ahead.

3. Use pull request to update the ontology:

Note that, by creating the pull request, you have triggered the automated CI/CD process that builds artefacts (documentation and OWL file). GitHub will show the current status of this process: whether it is still being processed, was successful, or has failed.

The generated artefacts should be verified to ensure they are OK. The minimum is to determine whether the CI/CD process completed successfully.

You may also manual inspection the generated artefacts, to verify the pull request has the desired effect. Even if you have already checked the commit locally (see step 1.e) checking the build artefacts provides a double-check.

The changes need to be review

The pull request is accepted in github's web interface. This will update the master branch to include the desired commit. This, in turn, will automatically close any corresponding issues.

If the pull request has a single commit then use the "Rebase and merge" option (the default). If there are multiple commits and you don't need to record the changes individually then use the "Squash and merge" option.

4. Clean up after ontology is updated

Ensure the local version of the master branch is up-to-date with the central github version:

git checkout master git pull --ff-only

Delete the branch that was just merged:

git branch -d ISS-51-add-term-tachyon-spectroscopy
